Bind ResourceQuota and LimitRange to a namespace

Creates name space label selectors and applies v1/ResourceQuota plus v1/LimitRange so admission enforces hard caps and default per-container limits.
PREREQUISITE
Admission plugin ResourceQuota enabled (default in most distros). Namespace name <ns> is empty or you have RBAC to create it.
YAML file namespace-quotas.yaml — field matrix:

ResourceQuota.spec.hard keys (CPU/memory/storage)

Key
Unit
Scoping note
requests.cpu
cores (e.g. 2, 500m)
Sum of pod spec.containers[].resources.requests.cpu
limits.memory
bytes / Ki/Mi/Gi
Sum of resources.limits.memory across all pods in ns
requests.storage
bytes
PVC spec.resources.requests.storage in class scope
podsdff
count
Max running pods in namespace
count/deployments.apps
count
Object count quota for apps/v1/Deployment
  1. Create or label namespace: kubectl create ns <ns>, optional kubectl label ns <ns> team=<id> cost-center=<code>
  2. Apply LimitRange in <ns> with spec.limits[].defaultRequest and default for Container type; example defaults: cpu: 100m, memory: 128Mi request; cpu: 500m, memory: 512Mi limit
    ADDITIONAL INFORMATION:
    LimitRangetype values: Container | Pod | PVC.
  3. Apply ResourceQuota with metadata.namespace: <ns> and spec.hard as in table; verify: kubectl describe quota -n <ns>
Result
Admission returns 403 Forbidden on pod create if requests exceed ResourceQuota; kubectl get events -n <ns> --field-selector type=Warning shows quota denial messages.
